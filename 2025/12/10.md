# わかったこと
- RESTは設計原則
- REST API 設計レベル
    - レベル0：HTTPを使っている
        - 単なる通信手段として利用。1URLで完結。
    - レベル1：リソースの概念を導入
        - リソースごとにURLを分割
        - HTTPメソッドは活用できていないので、GETかPOSTのみで通信
    - レベル2：レベル1に加えてHTTPメソッドを活用
    - レベル3：レベル2に加えてレスポンスにリソース感の繋がりが含まれる
        - レスポンスに現在の状態に関連するハイパーリンクが含まれる
- URI設計で考慮すること（URLはURIの一部）→巷に溢れているURIが似たり寄ったりなのはこの設計が基本的に根付いているから
    - 冗長性を排除
    - 人間が読んで理解できる
    - 大文字小文字が混在している
    - 単語はハイフンにする。アンダースコアはタイプライターで下線を引くためのものだった
    - 単語は複数形を利用する
    - エンコードを必要とする文字を使わない（URIから意味が理解できない）
    - サーバ側のアーキテクチャを反映しない
    - 改造しやすい（Hackable）システム依存の設計は意味が理解できない
    - ルールが統一されている
- クエリパラメータとパスパラメータ
    - クエリパラメータ：URLの末尾にある”?”に続くキーバリュー	【使用の判断基準】どのように取得したいか（条件・絞り込み・オプション）
    - パスパラメータ	URL：中に埋め込まれるパラメータ	【使用の判断基準】一意なリソースを表すのに必要かどうか
- HTTPメソッドとステータスコード
    - GET（データ取得）
・成功：①200 OK ②304 Not Modified
・失敗：①400 Bad Request ②401 Unauthorized ③403 Forbidden ④404 Not Found ⑤429 Too Many Requests ⑥500 Internal Sever error ⑦503 Service Unavailable（高負荷で応答不可）
    - POST（データ登録）
・成功：①200 OK ②201 Created ③202 Accepted
・失敗：①400 Bad Request ②401 Unauthorized ③403 Forbidden ④409 Conflict ⑤429 Too Many Requests ⑥500 Internal Sever error ⑦503 Service Unavailable
    - PUT（データ更新、データ登録）
・成功：①200 OK ②201 Created ③204 No Content
・失敗：①400 Bad Request ②401 Unauthorized ③403 Forbidden ④404 Not Found ⑤409 Conflict ⑥429 Too Many Requests ⑦500 Internal Sever error ⑧503 Service Unavailable
    - DELETE（データ削除）
・成功：①200 OK ②202 Accepted ③204 No Content（できるだけこのコードを返すようにする）
・失敗：①400 Bad Request ②401 Unauthorized ③403 Forbidden ④404 Not Found ⑤429 Too Many Requests ⑥500 Internal Sever error ⑦503 Service Unavailable
- データ内部構造の設計で考慮すること
    - エンベロープは使わない。ヘッダー情報と役割が被るのでエンベロープは使わない
    - オブジェクトはできるだけフラットにする
    - ページネーションをサポートする情報を返す（❌ “nextPage”: 1　⭕️“hasNext”：true, “nextPageToken”：”token name”）
    - プロパティの命令規則はAPI全体で統一する
    - 日付はRFC3339（W3C-DTF）形式を使う。インターネットで標準的に用いられるため
    - 大きな数値（64bit整数）は文字列で返す
- エラー表現で考慮する
    - エラー詳細はレスポンスボディに入れる。（400エラーなどは詳細が不明のため、エラー内容を追加する）
    - エラーの際にHTMLが返らないようにする
    - サービス閉塞時は”503”＋”Retry-After”でいつ再開するかの目処を記入する
- レートリミットで考慮すること（Fixed window、Sliding log、Sliding window［Fixed windowとsliding Logの良いとこどり］）
    - 誰に対して
    - 何に対して
    - 制限回数
    - 単位時間
        - アクセス制限が緩和される事例もある（そうした場面ごとに設定を変えられる）
